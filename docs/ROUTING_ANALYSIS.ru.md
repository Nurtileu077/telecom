## Почему OSRM тянет «слева и справа»

| Причина | Что видно на карте |
|--------|---------------------|
| **Каждый кабель маршрутизируется отдельно** | Два маршрута A→B и C→D по одной улице получают разные полосы движения |
| **OSRM = ось дороги / полоса** | Нет понятия «всегда тротуар слева»; алгоритм минимизирует время/длину |
| **Снэп `nearest` без стороны** | Точки попадают на разные рёбра графа |
| **Проход 2 без выравнивания** | Слияние по набору абонентов, а не по геометрии коридора |

### Что уже сделано в коде

1. **Сторона дороги** (`roadSide`: left / right / center) + отступ 3–6 м от оси — **Инструменты** и импорт.
2. **mergeParallel** перед проходом 2 — схлопывает параллельные OSRM-нити.
3. **Consolidation** — одна магистраль на сегмент улицы, муфты на развилках.
4. **Лестница ОК-4→8→12→16** от абонента к OLT.

**После смены стороны:** снова **① Проход 1** → **② Проход 2**.

## Чеклист сравнения с KMZ `gpon-network (11).kmz`

Откройте KMZ в Google Earth / QGIS и приложение рядом:

| # | Эталон (ожидание) | Проверить в приложении |
|---|-------------------|------------------------|
| 1 | Одна линия на проезжую часть (не две параллели) | После ①+②, сторона left/right |
| 2 | ОРКСП в центре группы камер | Центроид кластера |
| 3 | ОК-4 у дальних абонентов, рост к OLT | Подписи типов кабеля на схеме |
| 4 | Муфта на перекрёстке / развилке | Символ ⊕, не только на ОРК |
| 5 | Бокс у каждой камеры | Слой / сущность BOX-* |
| 6 | OLT → муфта ОК-16, муфта → ОРК ОК-4 | Фиксированные типы на L1 |
| 7 | Длины близки к смете (±10–15% запас) | Материалы / экспорт KMZ |

Типичные **недочёты эталона**, если он строился вручную или другим софтом:

- Дубли линий на одной улице (два KML LineString).
- Прямые отрезки через квартал (не по дорогам).
- Один тип ОК на всю сеть.
- ОРК не в геометрическом центре, а на первой камере.

## Варианты развития (перестраивать ли сайт целиком)

### A. Доработать текущий стек (рекомендуется сейчас)

**Плюсы:** быстро, уже есть топология Sergek, 2 прохода, экспорт.  
**Минусы:** публичный OSRM — лимиты, нет «настоящей» односторонней прокладки.

Дальше по приоритету:

1. Свой OSRM / Valhalla на сервере Казахстана (стабильность + `number=10` на nearest).
2. **Steiner tree / MST по графу улиц** — один раз построить граф из OSM, все кабели как пути в графе (гарантия 1 линия на ребро).
3. Импорт опор/колодцев из KMZ как обязательных узлов графа.

### B. Гибрид: OSM graph + наш UI

- Backend: Overpass → graph (osmnx / pgRouting).
- Прокладка: кратчайшие пути с **запретом повторного использования ребра** или с **общим коридором**.
- Front: оставить Leaflet + districts как сейчас.

**Плюсы:** максимально близко к «1 кабель = 1 дорога».  
**Минусы:** нужен сервер, 1–2 недели инженерии.

### C. Полная переписка

Имеет смысл только если нужны ГИС-уровень (версии проекта, СК-42, акты, интеграция 1С).  
Для Sergek GPON **не обязательно** — логика топологии важнее фреймворка.
